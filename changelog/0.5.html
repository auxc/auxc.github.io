The first tag experiment predictably wasn't very usable. Setting up tag relations inside the contents of each index entry proved to be hard to think about and unmaintainable even at a reasonably small index size.

The approaches the hierarchy as an emergent property of the index. The solution follows two simple rules:

  1. Display only the tags that are relevant to the currently ongoing search.
  2. Tag order in any UI where they are expected to be browsed defaults to sorting by the number of results a click would yield.

Maintenance is still a problem to be solved, but the goal is clear when editing an entry: keep the tags and descriptions accurate without worrying about the hierarchy.

Frontpage

  1. Relevant tags are now displayed at all times below the search bar to help browse the index by mouse and narrow down the search.
  2. Tags that would produce 0 results or no change are not shown.
